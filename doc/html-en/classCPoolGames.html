Member Enumeration Documentation

◆ t_algo

enum CPoolGames::t_algo

The algorithm of pool draws.

In addition to the algorithm itself, in each case, additional settings minDeposit, minSeconds are used, maxDeposit (only for AlgoTimeVol) for additional restrictions

See also
t_opts.minDeposit, t_opts.minSeconds, t_opts.maxDeposit
Enumerator
AlgoTime 

The probability of winning is proportional to the total time in the game.

The time of entry of each user into the draw is fixed. At the end of the game, all the time spent in the game of all users are summed up (and this sum is taken as probability 1.0). Next, a random number is normalized to this amount and the user is selected based on this number. As a result, the probability of a user winning is proportional to his time in the pool. Zeroing the deposit removes the user from the game, respectively, removes his time in the pool, this makes it pointless to exit and re-enter the game. This algorithm is more interesting for users with small deposits, the probability of winning does not depend on the size of the deposit, so you can invest little and win a lot.

See also
t_opts.minDeposit, t_opts.minSeconds, t_weight, t_game.weight
AlgoTimeVol 

The probability of winning is proportional to the sum of the products of the volume for the time in the game.

The time of entry of each user into the draw and his deposit at this moment is fixed. When replenishing the deposit, the previous deposit is multiplied by its time of presence in the pool in seconds and this amount is accumulated separately for each user. This allows users to increase the probability of winning by increasing the deposit during the game. At the end of the game, all the time spent in the game multiplied by the deposits of all users are summed up (taking into account deposits during the game, see above) (and this amount is taken as probability 1.0). Next, a random number is normalized to this amount and the user is selected based on this number. As a result, the probability of a user winning is proportional to the sum of the products of his deposit and the time spent in his liquidity pool. Zeroing the deposit removes the user from the game, respectively, removes his time in the pool, this makes it pointless to exit and re-enter the game. This algorithm is more interesting for users with large deposits, the probability of winning depends on the size of the deposit, so you can significantly increase the probability of winning by investing a large deposit.

See also
t_opts.minDeposit, t_opts.maxDeposit, t_weight, t_game.weight
AlgoEqual 

The probability of winning is equally likely.

The probability of winning is equally likely for all users in the pool who are present at the end of the game, all have a weight of 1. Using the algorithm without minSeconds is vulnerable to freeloaders who enter the pool only before the draw

See also
t_opts.minSeconds, t_weight, t_game.weight

◆ t_game_state

enum CPoolGames::t_game_state

States of the reward drawing game.

Enumerator
GameStateActivating 

The game is being launched.

This state remains in the vault only when users make or withdraw a deposit after the end of the draw in an empty pool. Launching a new drawing requires additional operations, it will be incorrect to demand gas consumption for these operations from the user. A full game launch will be made later by the pool manager.

GameStateActive 

There is a drawing game going on.

GameStateComplete 

The drawing game is over in time.

GameStateWaitRandom 

The drawing game is over in time, we are waiting for a random number to determine the winner.

GameStateWaitWinner 

The drawing game is over in time, a random number has been received (or is not required), we are waiting for the winner to be determined by an external code.

GameStatePause 

The games are suspended (the drawing game is completed, but the launch of the next one is blocked)

◆ t_pool_state

enum CPoolGames::t_pool_state

Pool status.

Enumerator
PoolStateActive 

The pool is active.

A reward is played periodically for all pool participants according to the pool settings

See also
StartPool, CreatePool
PoolStatePause 

Pool suspended.

If the drawing game is active, it continues until completion, but the next one will not be started.

See also
PausePool, CreatePool
PoolStateRemove 

Pool for deletion.

If the drawing game is active, it continues until the end. At the end of the game, when users have collected all deposits, the pool will be deleted during the last deposit debiting. If the pool is already empty at the time of setting the status, it will be deleted immediately.

See also
RemovePool
